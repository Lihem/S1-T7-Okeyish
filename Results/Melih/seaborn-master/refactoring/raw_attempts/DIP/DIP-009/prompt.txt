### Description
'FacetGrid' directly interacts with Matplotlib's global state and concrete figure/subplot creation functions.

### Reasoning
The `FacetGrid` class directly calls `plt.figure` and `fig.subplots` for figure and subplot creation. While Matplotlib itself is a dependency, `FacetGrid` could abstract the creation of its figure and axes, e.g., by receiving a figure factory or an axes manager through its constructor. This direct coupling to Matplotlib's concrete API and global state reduces flexibility and makes it harder to swap out the underlying plotting backend or mock during testing.

## Principle reference
**Dependency Inversion Principle** — High-level modules should not depend on low-level modules; both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions. When a class directly instantiates its dependencies instead of receiving them through injection, it violates DIP.

What typically signals a violation:
- Classes that directly instantiate their dependencies (using new/constructor calls)
- High-level business logic that imports and depends on low-level implementation details
- Missing dependency injection or inversion of control
- Hard-coded dependencies that make testing difficult
- Tight coupling between modules through concrete class references
