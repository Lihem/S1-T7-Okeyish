## Detected violation

- **Issue ID**: LSP-105
- **Principle**: Liskov Substitution Principle (LSP)
- **File**: `gson/src/main/java/com/google/gson/JsonElement.java`

### Description
Base class contract for 'getAsX' methods is too broad, forcing subclasses to throw IllegalStateException.

### Reasoning
The `JsonElement` abstract class defines numerous `getAsX()` methods (e.g., `getAsJsonObject()`, `getAsJsonArray()`, `getAsNumber()`, `getAsString()`). Its default implementations throw `IllegalStateException` if the element is not of the expected type. Subtypes like `JsonArray`, `JsonObject`, `JsonPrimitive`, and `JsonNull` cannot logically fulfill all these contracts. For example, a `JsonArray` cannot be converted to a `JsonObject` or a `Number`. This forces clients to perform type-checking (e.g., `isJsonArray()`) before calling `getAsJsonArray()`, which is a classic indicator of a 'fat interface' and an LSP violation. Subtypes are not substitutable for the base type without clients needing to know the concrete type to avoid runtime exceptions.

## Principle reference
**Liskov Substitution Principle** — Subtypes must be substitutable for their base types without altering the correctness of the program. When a subclass overrides behavior in a way that breaks the base class contract, it violates LSP.

What typically signals a violation:
- Subclasses that override methods and change their expected behavior
- Subclasses that raise exceptions not raised by the parent
- Subclasses that ignore or nullify parent class methods
- Type-checking (isinstance) used to handle subclasses differently
- Subclasses that violate preconditions or postconditions of parent methods
